Le probleme · 02DDDHEXAGONE

Quand la technique noie le metier

  • La regle "canicule = 3 jours > 35°C" est diluee dans des requetes SQL et des controleurs.
  • Changer de base ou d'API oblige a rouvrir le coeur metier.
  • Tester une regle exige de demarrer tout le framework.
  • Six mois plus tard, personne ne sait plus ou vit la definition d'une canicule.

L'architecture hexagonale inverse la dependance : la technique s'adapte au metier, jamais l'inverse. Le DDD, lui, dit comment ecrire ce metier.

Le metier au centre. La technique branchee autour.

Fondations · 03DDD

Le DDD, c'est quoi exactement ?

Domain-Driven Design

Une facon de concevoir un logiciel dans laquelle le modele du metier est le coeur du systeme, et non un detail range entre la base de donnees et l'ecran.

Terme popularise par Eric Evans en 2003. "Domaine" = le sujet dont parle le logiciel. Ici : la meteorologie et les vagues de chaleur.

Le pari du DDD : dans une application non triviale, la difficulte n'est pas technique. Elle est de comprendre le metier, puis de garder cette comprehension lisible dans le code pendant des annees.

Ce que le DDD n'est pas

  • Ni une bibliotheque, ni un framework. Le paquet fr.meteo.surveillance.domaine n'a aucune dependance : ni Spring, ni JPA, ni Jackson.
  • Ni une arborescence de dossiers. Nommer un paquet domaine ne suffit pas : c'est ArchitectureHexagonaleTest qui le prouve, a chaque build.
  • Ni reserve aux gros projets. Mais inutile sur un CRUD sans regles.
  • Ni l'architecture hexagonale. Les deux se marient bien, mais repondent a des questions differentes.

Brique tactique · 05DDD

Objet-valeur : la theorie

Value object

Un objet qu'on identifie par ce qu'il vaut, pas par qui il est. Deux exemplaires de meme valeur sont interchangeables.

  • Immuable : aucun setter. Toute "modification" renvoie une nouvelle instance. Donc partageable sans risque.
  • Sans identite : on redefinit equals sur la valeur. 25°C est egal a 77°F : c'est la meme chaleur.
  • Auto-validant : le constructeur refuse l'absurde. Une temperature sous le zero absolu n'existe pas.
  • Porteur de comportement : convertir et comparer sont des regles metier ; elles vivent ici, pas dans un TemperatureUtils.

Analogie

Un billet de 10 € est un objet-valeur : si l'on echange le votre contre le mien, rien n'a change. Votre compte en banque, lui, a une identite : on ne l'echange pas contre un autre de meme solde.

fr/meteo/surveillance/domaine/modele/Temperature.java

public final class Temperature {

  private static final double ZERO_ABSOLU_CELSIUS = -273.15;

  private final double valeur;          // champs finaux
  private final UniteTemperature unite;

  private Temperature(double valeur, UniteTemperature unite) {
    // auto-validation : l'objet ne peut pas naitre invalide
    if (convertirEnCelsius(valeur, unite) < ZERO_ABSOLU_CELSIUS) {
      throw new IllegalArgumentException(
          "Temperature impossible : sous le zero absolu");
    }
    this.valeur = valeur;
    this.unite = unite;
  }
}

Brique tactique · 06DDD

Objet-valeur : le code

Trois details qui font la difference entre une classe et un vrai objet-valeur.

  1. Fabriques nommees plutot qu'un constructeur public : Temperature.celsius(35) dit son unite. Impossible de creer une temperature "sans unite".
  2. Une methode = une regle metier. atteintOuDepasse n'est pas un utilitaire : c'est la phrase que prononce le meteorologue.
  3. equals sur la valeur reelle, en Celsius, avec une tolerance qui absorbe les arrondis de conversion.

Verifie par TemperatureTest : Temperature.celsius(25) est equals a Temperature.fahrenheit(77).

fr/meteo/surveillance/domaine/modele/Temperature.java (suite)

// 1. fabriques : le nom porte l'unite
public static Temperature celsius(double v) {
  return new Temperature(v, UniteTemperature.CELSIUS);
}
public static Temperature fahrenheit(double v) {
  return new Temperature(v, UniteTemperature.FAHRENHEIT);
}

// 2. du comportement metier, pas un helper technique
public boolean atteintOuDepasse(Temperature seuil) {
  return this.valeurEnCelsius() >= seuil.valeurEnCelsius();
}
public boolean estPlusChaudeQue(Temperature autre) {
  return this.valeurEnCelsius() > autre.valeurEnCelsius();
}

// 3. egalite par valeur : 25 C et 77 F sont le meme objet
@Override
public boolean equals(Object o) {
  if (!(o instanceof Temperature autre)) return false;
  return Math.abs(this.valeurEnCelsius()
                - autre.valeurEnCelsius()) < 1e-4;
}

Brique tactique · 07DDD

Le record, objet-valeur pret a l'emploi

record · Java 16+

Un type dont le compilateur genere le constructeur, les accesseurs, equals, hashCode et toString a partir de la seule liste des champs. Ses champs sont finaux : il est immuable par construction.

Le record offre gratuitement trois des quatre proprietes de la slide precedente : immuabilite, egalite par valeur, hashCode coherent. Il ne reste qu'a ecrire la quatrieme, la validation.

  • Le constructeur compact — public Localisation {, sans parametres ni parentheses. Il s'execute avant l'affectation des champs : c'est la qu'on refuse l'absurde.
  • Emballer une valeur unique — StationId n'ajoute aucun champ a un UUID. Il ajoute du sens, et de la securite de type. On verra slide 20 ce que cela evite.

Consequence : une Localisation qui existe a forcement une ville. Plus jamais de verification defensive ailleurs dans le code.

fr/meteo/surveillance/domaine/modele/Localisation.java

public record Localisation(String ville, String codePostal) {

  // constructeur compact : la validation, et rien d'autre
  public Localisation {
    if (ville == null || ville.isBlank()) {
      throw new IllegalArgumentException("La ville est obligatoire");
    }
    if (codePostal == null || codePostal.isBlank()) {
      throw new IllegalArgumentException("Le code postal est obligatoire");
    }
  }
}

// egalite par valeur, generee : rien a ecrire
new Localisation("Lyon", "69000")
    .equals(new Localisation("Lyon", "69000"));   // true

fr/meteo/surveillance/domaine/modele/StationId.java

public record StationId(UUID valeur) {

  public StationId {
    Objects.requireNonNull(valeur, "L'identifiant est obligatoire");

Strategique · 08DDD

Contexte delimite : la question a poser

Bounded context

Une zone du logiciel a l'interieur de laquelle chaque mot du metier a un sens unique. Le mot change de sens quand on franchit la frontiere.

Le test concret n'est pas "combien de microservices ?" mais : le meme mot designe-t-il deux modeles differents selon a qui je parle ?

Posons la question sur ce projet. Le mot est "station".

  • Pour le previsionniste : une station est une source de releves de temperature. Il veut savoir s'il y a canicule.
  • Pour le technicien de maintenance : une station est un boitier avec un numero de serie, qu'il faut etalonner chaque annee. La temperature ne l'interesse pas.

Strategique · 09DDD

Le meme mot, deux modeles

Voici les deux classes, cote a cote. Elles decrivent le meme objet du monde reel : le boitier installe a Lyon. Elles ne partagent pas une ligne.

◉ contexte surveillance

fr/meteo/surveillance/domaine/modele/StationMeteo.java

public class StationMeteo {

  private final StationId id;
  private final Localisation localisation;
  private final List<ReleveTemperature> releves;

  // l'invariant : un seul releve par jour
  public void enregistrer(ReleveTemperature releve) {
    boolean jourDejaConnu = releves.stream()
        .anyMatch(r -> r.jour().equals(releve.jour()));
    if (jourDejaConnu) {
      throw new ReleveDejaPresentException(id, releve.jour());
    }
    releves.add(releve);
  }
}

◉ contexte maintenance

fr/meteo/maintenance/domaine/modele/Station.java

public class Station {

  public static final int VALIDITE_ETALONNAGE_JOURS = 365;

  private final StationId id;
  private final NumeroSerie numeroSerie;
  private EtatOperationnel etat;
  private LocalDate dernierEtalonnage;

  // un autre invariant, sur le meme objet reel
  public void mettreEnService(LocalDate aujourdHui) {
    if (etalonnagePerime(aujourdHui)) {
      throw new EtalonnagePerimeException(id, dernierEtalonnage);
    }
    this.etat = EtatOperationnel.EN_SERVICE;
  }
}

Aucune classe n'est "la vraie station". Chercher a les fusionner produirait une classe qui parle a la fois de canicules et de banc de calibration : le fourre-tout que le DDD cherche precisement a eviter.

Strategique · 10DDD

Chacun son identifiant

Le point le plus contre-intuitif du decoupage. Il existe deux classes StationId dans ce depot, presque identiques.

Ce n'est pas un oubli de factorisation. Si la maintenance importait le StationId de la surveillance, les deux modeles seraient soudes : changer le type de l'identifiant d'un cote casserait l'autre.

La regle

On factorise a l'interieur d'un contexte. Jamais entre deux contextes.

Les contextes ne partagent que la valeur brute — ici un UUID — qui sert d'identifiant de correlation.

fr/meteo/surveillance/domaine/modele/StationId.java

package fr.meteo.surveillance.domaine.modele;

public record StationId(UUID valeur) { ... }

fr/meteo/maintenance/domaine/modele/StationId.java

package fr.meteo.maintenance.domaine.modele;

// Duplication DELIBEREE, pas un manque de factorisation.
public record StationId(UUID valeur) { ... }

La frontiere est verifiee a chaque build, dans les deux sens

Strategique · 11DDD

L'arborescence est une affirmation

Un paquet domaine pose a la racine affirme qu'il existe une seule lecture legitime du metier. Tout le reste devient, implicitement, un module secondaire.

Si deux contextes sont freres dans le metier, ils doivent etre freres dans l'arborescence. Sinon le code contredit le cours.

  • Les contextes sont des paquets de premier niveau : surveillance et maintenance, cote a cote.
  • Au-dessus d'eux, uniquement ce qui n'appartient a aucun : le demarrage Spring Boot, la racine de composition, les tests d'architecture.
  • Chaque contexte a la meme forme interne. La maintenance a son hexagone complet, jusqu'a sa table et son controleur. Simplement plus petit : aucune regle n'y echappe encore a l'agregat, donc pas de service/.

Le test : peut-on nommer le contexte d'une classe rien qu'en lisant son chemin ? Si non, l'arborescence ment.

Strategique · 12DDDHEXAGONE

Piege : une couche n'est pas un contexte

Le depot contient quatre classes qui parlent de stations. Trois d'entre elles ne forment qu'un seul contexte. Savoir les distinguer evite de crier "bounded context" a chaque DTO.

Un modele, trois representations

StationMeteole modele, dans le domaine
StationMeteoJpasa forme en base
StationReponsesa forme en JSON

StationMeteoMapper traduit entre les deux premieres sans rien perdre ni rien inventer. Meme contexte.

Un autre modele

Stationle modele du contexte maintenance

Aucun mapper ne peut la produire depuis StationMeteo : il faudrait jeter les releves et inventer un numero de serie et une date d'etalonnage.

Le test de la traduction

Puis-je traduire d'un type vers l'autre sans rien perdre ni rien inventer ? Oui → ce sont deux representations d'un meme modele, donc un seul contexte. Non → ce sont deux modeles, donc deux contextes.

Les trois slides suivantes appliquent ce test au code : d'abord une traduction qui marche, ensuite une qui ne peut pas exister.

Strategique · 13HEXAGONE

Un modele, trois representations

Suivons une station de surveillance a travers les couches de son hexagone. A chaque frontiere technique elle change de forme, jamais de sens : le mapper ne perd rien et n'invente rien. C'est ce qui prouve qu'on reste dans un seul contexte.

StationReponse

  2. La forme en base — StationMeteoJpa. Mutable, constructeur vide : les exigences de l'ORM.
  3. La forme en JSON — StationReponse, aplatie pour le client HTTP.
Le critere

L'aller-retour versJpa puis versDomaine rend une station equivalente a celle du depart.

Rien ne s'evapore, rien n'apparait. Trois formes, un seul modele — donc un seul contexte.

Strategique · 14HEXAGONE

La traduction, dans le code

Le mapper vit dans l'adaptateur, jamais dans le domaine. C'est lui qui absorbe les exigences de l'ORM pour que l'agregat n'ait pas a les connaitre.

  • Aller — versJpa aplatit : StationId devient un UUID, Localisation devient deux colonnes.
  • Retour — versDomaine passe par la fabrique reconstituer. Les invariants sont revalides au chargement.
  • Consequence — une ligne corrompue en base ne peut pas ressusciter un agregat invalide. Elle leve une exception du domaine.

Le DTO, lui, ne fait que sortir : aucun versDomaine ne remonte depuis un StationReponse. Une entree HTTP passe par un objet de commande (slide 35).

surveillance/adaptateur/sortant/persistance/StationMeteoMapper.java

// DOMAINE -> PERSISTANCE : on aplatit, on ne juge pas
static StationMeteoJpa versJpa(StationMeteo station) {
  return new StationMeteoJpa(
      station.id().valeur(),              // StationId -> UUID
      station.localisation().ville(),     // Localisation
      station.localisation().codePostal(),//  -> 2 colonnes
      relevesJpa);
}

// PERSISTANCE -> DOMAINE : on reconstruit un agregat VALIDE
static StationMeteo versDomaine(StationMeteoJpa jpa) {
  return StationMeteo.reconstituer(
      new StationId(jpa.getId()),
      new Localisation(jpa.getVille(), jpa.getCodePostal()),
      releves);
}

Strategique · 15DDD

Ce qu'aucun mapper ne peut faire

Le contexte maintenance a maintenant, lui aussi, son entite JPA et son DTO. Essayons d'ecrire le mapper qui transformerait une StationMeteo en StationJpa de maintenance. Il faudrait jeter des donnees et en inventer d'autres. Ce n'est pas une traduction : c'est une autre lecture du monde.

tablecolonnes
stationsid, ville, code_postal, releves[]
stations_maintenanceid, numero_serie, etat, dernier_etalonnage
  • Une seule colonne commune : id. Et encore : c'est la valeur brute de l'UUID, pas une cle etrangere entre modeles.
  • Aucune conversion possible : d'ou viendrait le numero de serie ? Que deviendraient les releves ?
  • Deux tables, deux depots, deux mappers. Rien n'est mutualise : c'est ce qui leur permet d'evoluer separement.

La frontiere va jusqu'au bord du systeme : deux URL, deux JSON, zero champ commun hors identifiant.

Langage · 16DDD

Le langage omnipresent

Ubiquitous language

Un seul vocabulaire, partage par les experts metier et par le code, a l'interieur d'un contexte. Le mot dit par le meteorologue est le mot lu dans la classe Java.

"Omnipresent" parce qu'il vaut partout : dans la reunion, dans le ticket, dans le nom de la classe, dans le message d'erreur, dans le test.

Le benefice n'est pas cosmetique. Chaque traduction mentale ("le type 0, c'est bien Celsius ?") est une occasion de se tromper.

Le projet pousse le principe jusqu'a l'API : ReleveRequete exige un champ unite valant CELSIUS ou FAHRENHEIT. Un client HTTP ne peut pas envoyer une temperature sans unite.

Le piege n°1 · 18DDD

Le modele anemique

Depuis la slide 05 on manipule des objets-valeurs sans avoir dit contre quoi ils nous protegent. Le voici.

Anemic domain model

Une classe reduite a des champs, des getters et des setters. Elle porte des donnees, aucune regle.

Le comportement metier part alors vivre ailleurs, en general dans un "Service" fourre-tout.

Ce n'est pas un bug : cela compile, cela passe les tests. Cela se paie plus tard, le jour ou la meme regle existe en trois exemplaires legerement differents — et ou l'un des trois est faux.

✗ anemique · un sac de donnees

class Temperature {
  double valeur;   // 25 ? En Celsius ? En Fahrenheit ?
  void setValeur(double v) { this.valeur = v; }
  double getValeur() { return valeur; }
}

// la regle metier finit dupliquee, loin du modele
if (t1.getValeur() >= 35) { ... }  // dans le controleur
if (t2.getValeur() >= 35) { ... }  // et dans le batch

✓ riche · la donnee ET la regle

public final class Temperature {
  private final double valeur;
  private final UniteTemperature unite;

  public boolean atteintOuDepasse(Temperature seuil) {
    return valeurEnCelsius() >= seuil.valeurEnCelsius();
  }
}

// un seul endroit fait autorite ; l'appel se lit
releve.temperature().atteintOuDepasse(seuil);

Le piege n°1 · 19DDD

Pourquoi on a commence par les objets-valeurs

Chaque objet-valeur croise depuis la slide 05 existe pour rendre une phrase impossible a ecrire. Mis bout a bout, ils forment un modele qui ne peut pas devenir anemique : la regle n'a nulle part ailleurs ou aller.

Objet-valeurRend impossibleLe mecanisme

Localisationune station sans villeconstructeur compact validant
StationIdpasser un identifiant de canicule la ou on attend une stationtype dedie, verifie a la compilation

La regle de pouce

Si vous ecrivez objet.getX() pour decider quelque chose a propos de cet objet, la decision appartient sans doute a l'objet.

Un getter n'est pas interdit. Il devient suspect des qu'il sert a prendre, a l'exterieur, une decision que l'objet pourrait prendre lui-meme.

CaniculeReponse.depuis(c) appelle six getters : c'est legitime, il traduit vers le monde exterieur, il ne decide rien.

Brique tactique · 20DDD

Typer jusqu'aux identifiants

Primitive obsession

Representer un concept metier par un type technique nu (String, UUID, int). Le compilateur ne peut alors plus rien verifier pour vous.

Un UUID de station et un UUID de canicule ont exactement le meme type. Les intervertir compile parfaitement, et se decouvre en production.

Brique tactique · 22DDD

Agregat : la theorie

Aggregate · aggregate root

Une grappe d'objets qu'on charge, modifie et sauvegarde comme un tout. Un seul objet de la grappe est visible de l'exterieur : la racine.

La racine est la frontiere de coherence : elle seule laisse entrer une modification, donc elle seule peut garantir les regles.

Invariant

Une regle qui doit toujours etre vraie, a chaque instant. Ici : "une station a au plus un releve par jour".

Les quatre regles de conception

  1. Un agregat protege un invariant reel. Pas de grappe "parce que c'est pratique" : s'il n'y a rien a garantir, il n'y a pas d'agregat.
  2. Le plus petit possible. Une grosse grappe se charge lentement et se verrouille souvent.
  3. Un agregat en reference un autre par son identite, jamais par un pointeur direct sur l'objet.
  4. Une transaction ne modifie qu'un agregat. Les autres suivent, plus tard.

Regle 4, dans le code : ServiceDetectionCanicule charge une StationMeteo mais ne sauvegarde que des Canicule. Les deux agregats ne sont jamais ecrits ensemble.

Brique tactique · 23DDD

Agregat : garder l'invariant

Deux mecanismes, et ils sont indissociables. Le premier verifie la regle ; le second empeche de le contourner.

  • Une methode metier nommee — enregistrer(releve), et non getReleves().add(...). C'est le seul chemin d'entree.
  • Une collection encapsulee — l'exterieur ne recoit qu'une copie non modifiable. Sans cela, le premier mecanisme ne sert a rien.
  • Une exception du metier — ReleveDejaPresentException vit dans le domaine : c'est un evenement du langage, pas une panne technique.

Les quatre tests de StationMeteoTest verrouillent exactement ces trois points, dont la_liste_exposee_est_non_modifiable.

fr/meteo/surveillance/domaine/modele/StationMeteo.java

public class StationMeteo {           // racine d'agregat

  private final StationId id;
  private final Localisation localisation;
  private final List<ReleveTemperature> releves;  // interne

  // seul chemin pour ajouter un releve : l'invariant tient
  public void enregistrer(ReleveTemperature releve) {
    boolean jourDejaConnu = releves.stream()
        .anyMatch(r -> r.jour().equals(releve.jour()));
    if (jourDejaConnu) {
      throw new ReleveDejaPresentException(id, releve.jour());
    }
    releves.add(releve);
  }

  // copie NON MODIFIABLE : impossible de contourner la regle
  public List<ReleveTemperature> releves() {
    return releves.stream()
        .sorted(Comparator.comparing(ReleveTemperature::jour))
        .toList();
  }

  // l'identite, et elle seule, definit l'egalite
  @Override
  public boolean equals(Object o) {
    return o instanceof StationMeteo autre && id.equals(autre.id);
  }
}

Brique tactique · 26DDD

La fabrique

Factory

Une methode qui sait construire un objet du domaine deja valide et complet, a partir de ce que l'appelant possede vraiment.

Le constructeur de Canicule est prive. On ne lui passe donc jamais une date de fin ni un pic : ce serait donner a l'appelant l'occasion de les calculer faux.

// fabrique nommee : le nom dit dans quel cas on l'emploie
public static Canicule surPeriode(StationId stationId,
    Localisation localisation, List<ReleveTemperature> releves) {

  if (releves == null || releves.isEmpty()) {
    throw new IllegalArgumentException(
        "Une canicule couvre au moins un releve");
  }

  // le domaine calcule, l'appelant ne fournit pas
  LocalDate debut = releves.getFirst().jour();
  LocalDate fin   = releves.getLast().jour();
  Temperature pic = releves.stream()
      .map(ReleveTemperature::temperature)
      .reduce((a, b) -> a.estPlusChaudeQue(b) ? a : b)
      .orElseThrow();

  return new Canicule(CaniculeId.nouveau(), stationId,
                      localisation, debut, fin, pic);
}

Brique tactique · 29DDD

Service de domaine

Detecter les canicules ne rentre naturellement dans aucun agregat : il faut raisonner sur une serie de releves, et en produire de nouveaux agregats Canicule.

Quand une regle n'a pas de proprietaire evident, on ne la force pas dans un agregat : on cree un service de domaine.

  • Sans etat : aucun champ mutable, donc trivialement testable.
  • Vocabulaire 100% metier : seContinue, cloturer.
  • Pas un @Service Spring : du Java pur, executable dans un simple main().

Les 5 tests de DetecteurCaniculeTest tournent en 22 ms, sans contexte Spring.

fr/meteo/surveillance/domaine/service/DetecteurCanicule.java

public class DetecteurCanicule {   // aucune annotation

  public List<Canicule> detecter(StationMeteo station,
                                 SeuilCanicule seuil) {
    List<Canicule> canicules = new ArrayList<>();
    List<ReleveTemperature> serieEnCours = new ArrayList<>();

    for (ReleveTemperature releve : station.releves()) {
      if (seuil.estFranchiPar(releve)
          && seContinue(serieEnCours, releve)) {
        serieEnCours.add(releve);
      } else {
        cloturer(serieEnCours, seuil, station, canicules);
        serieEnCours = new ArrayList<>();
        if (seuil.estFranchiPar(releve)) serieEnCours.add(releve);
      }
    }
    cloturer(serieEnCours, seuil, station, canicules);
    return canicules;
  }

  // un releve prolonge la serie s'il tombe le lendemain
  private boolean seContinue(List<ReleveTemperature> serie,
                             ReleveTemperature releve) {
    if (serie.isEmpty()) return true;
    return releve.jour().equals(serie.getLast().jour().plusDays(1));
  }
}

Il met en scene un cas d'usage, toujours selon le meme rythme en trois temps. Notez ce qu'il ne fait pas : il ne calcule aucune regle.

Architecture · 32HEXAGONE

L'hexagone

La regle de dependance

Le code d'une couche interne ne connait jamais le nom d'une classe d'une couche externe. Toutes les fleches de compilation pointent vers le centre.

Pourquoi un hexagone ? Pour rien de precis : les six cotes suggerent seulement qu'on peut brancher plusieurs facons d'entrer et de sortir. Le dessin aurait pu etre un cercle.

Ce qui compte : le coeur ne depend de rien. Tout ce qui est technique est a la peripherie et depend du centre.

Chaque contexte delimite a son propre hexagone : la maintenance a ses ports (ConsulterStation, DepotStation), son adaptateur JPA et son controleur.

Les deux hexagones de ce projet

Deux familles de ports

Un port est une interface Java, posee dans le coeur. Un adaptateur est la classe technique qui la branche au monde reel. Le sens de l'appel distingue les deux familles.

Ports · 36HEXAGONE

L'inversion de dependance

Dependency inversion

L'interface appartient a celui qui appelle, pas a celui qui implemente. Le domaine declare le contrat ; la technique s'y plie.

C'est le mecanisme exact qui rend l'hexagone possible. Sans lui, le domaine devrait importer l'adaptateur pour l'appeler.

La fleche de compilation pointe vers le domaine : c'est l'adaptateur qui importe l'interface. La fleche d'execution en sort : au runtime, le domaine appelle bien la base.

Consequence concrete : remplacer H2 par un fichier texte ne touche pas une ligne du domaine.

Piege classique · 39HEXAGONE

Le domaine ignore HTTP

Le domaine leve des exceptions au vocabulaire metier : ReleveDejaPresentException, StationIntrouvableException. Il ne sait pas, et ne doit pas savoir, qu'elles deviendront un 409 ou un 404.

La correspondance est un detail d'adaptateur. Elle vit dans une seule classe de la couche web.

  • ReleveDejaPresentException → 409 Conflict : l'invariant de l'agregat a ete viole.
  • StationIntrouvableException → 404 : erreur d'orchestration, levee par la couche application.
  • IllegalArgumentException → 400 : le domaine a refuse une donnee absurde, par exemple sous le zero absolu.

  @ExceptionHandler(StationMeteo.ReleveDejaPresentException.class)
  public ProblemDetail releveDejaPresent(
      StationMeteo.ReleveDejaPresentException ex) {
    return ProblemDetail.forStatusAndDetail(
        HttpStatus.CONFLICT, ex.getMessage());

  // station demandee inexistante -> 404 Not Found
  @ExceptionHandler(StationIntrouvableException.class)

      StationIntrouvableException ex) {
    return ProblemDetail.forStatusAndDetail(

  // donnee refusee par le domaine (ex : zero absolu) -> 400
  @ExceptionHandler(IllegalArgumentException.class)
  public ProblemDetail donneeInvalide(IllegalArgumentException ex) {

La regle, rendue executable

Toutes ces regles ne tiennent que par la discipline. Or la discipline s'erode : un import de trop, un vendredi soir, et le domaine depend de Spring.

ArchUnit inspecte les dependances entre paquets et casse le build si quelqu'un franchit une frontiere. La documentation ne peut plus mentir : elle est compilee.

30 tests verts dont 7 d'architecture

Les 7 regles se lisent en deux familles : la regle de dependance a l'interieur d'un contexte (4 regles), et la frontiere de contexte entre les deux (3 regles, dont celle qui protege la racine de composition).

test/architecture/ArchitectureHexagonaleTest.java

// le coeur des DEUX contextes est du Java pur
@Test
void aucun_domaine_ne_depend_d_un_framework() {
  ArchRule regle = noClasses()
      .that().resideInAnyPackage(
          "fr.meteo.surveillance.domaine..",
          "fr.meteo.maintenance.domaine..")
      .should().dependOnClassesThat().resideInAnyPackage(FRAMEWORKS)
      .because("le coeur metier de chaque contexte doit rester "
             + "du Java pur");

  regle.check(classes);
}

// la racine de composition connait les contextes, jamais l'inverse
@Test
void aucun_contexte_ne_depend_de_la_racine_de_composition() {
  noClasses().that().resideInAnyPackage(SURVEILLANCE, MAINTENANCE)
    .should().dependOnClassesThat().resideInAPackage("fr.meteo.config..")
    .check(classes);
}

Ce qui bouge

  • Le controleur : REST aujourd'hui, un message Kafka demain.
  • La base : H2 en demo, PostgreSQL en production.
  • L'alerte : NotificateurCaniculeLog ici, un SMS ensuite.

Ce qui ne bouge pas

  • La definition d'une canicule.
  • L'invariant "un releve par jour".
  • Les 14 tests du domaine surveillance, qui tournent en millisecondes.

A retenir · 42DDDHEXAGONE

Six idees a emporter

  1. Cherchez d'abord ou le vocabulaire change de sens. C'est la que passe la frontiere d'un contexte.
  2. Nommez les choses comme le metier les nomme. Tout le reste en decoule.
  3. Mettez la regle a cote de la donnee. Un objet qui expose ses champs pour qu'on decide a sa place est anemique.
  4. Un agregat existe pour proteger un invariant. S'il n'y a rien a proteger, il n'y a pas d'agregat.
  5. L'interface appartient a l'appelant. C'est l'inversion de dependance, et c'est tout l'hexagone.
  6. Rendez la regle d'architecture executable. Une convention non testee est deja violee.

Le metier au centre. La technique branchee autour.

Le controleur, la base et l'alerte sont interchangeables. Le coeur, lui, ne bouge pas. C'est toute la promesse de l'hexagone — et le DDD est ce qui rend ce coeur digne d'etre protege.
